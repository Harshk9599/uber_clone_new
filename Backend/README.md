# Backend API ‚Äî Users

## POST /users/register

Registers a new user and returns an authentication token and the created user object.

- **Endpoint:** `POST /users/register`
- **Headers:** `Content-Type: application/json`
- **Success Status:** `201 Created`

### Description

Creates a new user in the database. The endpoint validates the incoming payload and hashes the password before storing. On success it returns a JWT `token` and the created `user` (password is not returned).

### Request Body

Content type: `application/json`

Required fields:

- `fullname.firstname` (string) ‚Äî required, minimum 3 characters.
- `email` (string) ‚Äî required, must be a valid email address.
- `password` (string) ‚Äî required, minimum 6 characters.

Optional fields:

- `fullname.lastname` (string) ‚Äî optional, recommended minimum 3 characters.

Example request:

```json
{
  "fullname": {
    "firstname": "Jane",
    "lastname": "Doe"
  },
  "email": "jane.doe@example.com",
  "password": "s3cr3tpass"
}
```

### Responses

- `201 Created` ‚Äî User created successfully.

  Example success response:

  ```json
  {
    "token": "<jwt_token_here>",
    "user": {
      "_id": "609d9f9f9f9f9f9f9f9f9f9f",
      "fullname": {
        "firstname": "Jane",
        "lastname": "Doe"
      },
      "email": "jane.doe@example.com",
      "socketId": null
    }
  }
  ```

- `400 Bad Request` ‚Äî Validation failed (missing/invalid fields). Response contains an `errors` array produced by `express-validator`.

  Example:

  ```json
  {
    "errors": [
      { "msg": "Invalid Email Address", "param": "email", "location": "body" },
      { "msg": "Password must be at least 6 characters long", "param": "password", "location": "body" }
    ]
  }
  ```

- `409 Conflict` ‚Äî Email already exists (may be returned by the database). Implementation may return a `500` depending on error handling; consider adding explicit duplicate-check handling in the service for a consistent `409` response.

- `500 Internal Server Error` ‚Äî Unexpected errors.

### Notes

- The server hashes passwords using bcrypt before saving. The controller calls `userModel.hashPassword()` and the model stores the hashed password (the `password` field is excluded by default when returning documents).
- Validation rules are applied in the route using `express-validator` (see `routes/user.routes.js`).

## POST /users/login

Authenticates an existing user and returns a JWT token plus the user object.

- **Endpoint:** `POST /users/login`
- **Headers:** `Content-Type: application/json`
- **Success Status:** `201 Created`

### Description

Validates the login payload, finds the user by email, checks the password, and returns a token on success.

### Request Body

Content type: `application/json`

Required fields:

- `email` (string) ‚Äî required, must be a valid email address.
- `password` (string) ‚Äî required, minimum 6 characters.

Example request:

```json
{
  "email": "jane.doe@example.com",
  "password": "s3cr3tpass"
}
```

### Responses

- `201 Created` ‚Äî Login successful.

  Example success response:

  ```json
  {
    "token": "<jwt_token_here>",
    "user": {
      "_id": "609d9f9f9f9f9f9f9f9f9f9f",
      "fullname": {
        "firstname": "Jane",
        "lastname": "Doe"
      },
      "email": "jane.doe@example.com",
      "socketId": null
    }
  }
  ```

- `400 Bad Request` ‚Äî Validation failed (missing/invalid fields). Response contains an `errors` array produced by `express-validator`.

- `401 Unauthorized` ‚Äî Invalid email or password.

  Example:

  ```json
  {
    "message": "Invalid email or password"
  }
  ```

- `500 Internal Server Error` ‚Äî Unexpected errors.

### Notes

- The login route uses `userModel.comparePassword()` to verify the submitted password against the stored hashed password.
- After successful authentication, the response includes a JWT token generated by `user.generateAuthToken()`.

## GET /users/profile

Returns the authenticated user's profile information.

- **Endpoint:** `GET /users/profile`
- **Headers:** `Cookie: token=<jwt>` or `Authorization: Bearer <jwt>`
- **Success Status:** `200 OK`

### Description

Requires a valid JWT (sent via cookie or `Authorization` header). The `authUser` middleware verifies the token and sets `req.user`. The endpoint returns the user document for the authenticated user.

### Responses

- `200 OK` ‚Äî Returns the authenticated user's profile.

  Example success response:

  ```json
  {
    "_id": "609d9f9f9f9f9f9f9f9f9f9f",
    "fullname": {
      "firstname": "Jane",
      "lastname": "Doe"
    },
    "email": "jane.doe@example.com",
    "socketId": null
  }
  ```

- `401 Unauthorized` ‚Äî Missing/invalid token or blacklisted token.

- `500 Internal Server Error` ‚Äî Unexpected errors.

### Notes

- The route is protected by `auth.middleware` (`authUser`) which checks the token and ensures it is not blacklisted (see `middlewares/auth.middleware.js`).

## GET /users/logout

Logs the user out by clearing the auth cookie and blacklisting the token.

- **Endpoint:** `GET /users/logout`
- **Headers:** `Cookie: token=<jwt>` or `Authorization: Bearer <jwt>`
- **Success Status:** `200 OK`

### Description

Clears the `token` cookie and writes the token to a blacklist collection so it cannot be reused until the token expires or the blacklist TTL removes it.

### Responses

- `200 OK` ‚Äî Logout successful. Example:

  ```json
  { "message": "logged out successfully" }
  ```

- `401 Unauthorized` ‚Äî Missing/invalid token.

- `500 Internal Server Error` ‚Äî Unexpected errors.

### Notes

- The controller clears the cookie and stores the token in `blacklistToken.model.js`. The blacklist uses a TTL index so blacklisted tokens expire automatically.
